COLA 架构

由阿里巴巴开源的应用架构，它深度集成了领域驱动设计和整洁架构的思想

旨在解决传统 MVC 架构在复杂业务场景下代码混乱、职责不清、难以维护的问题

核心思想

以业务为核心，实现关注点分离

  1. 依赖倒置原则：如图所示，所有依赖都指向中心的领域层（Domain）。业务逻辑不依赖于任何技术实现，而是技术实现来依赖业务逻辑

    ​ 这保证了业务核心的稳定性和纯粹性

  2. 业务逻辑内聚：将原本散落在 Service 中的业务逻辑，收敛到领域层的领域对象（DO）中，让代码职责更清晰

  3. 命令查询职责分离 (CQRS)：将“改变系统状态”的命令（Command）操作和“不改变状态”的查询（Query）操作分离开，让系统意图更明确

核心概念

概念 全称 所在分层 用途 特点
DTO Data Transfer Object client 数据传输对象 用于应用层与外部（如前端）或系统间（如RPC）的数据交互，是系统的门面
DO Domain Object domain 领域对象 业务的核心。封装了业务属性和行为（方法），是富血模型
PO Persistent Object infrastructure 持久化对象 与数据库表结构一一对应，用于数据的持久化操作，是贫血模型

其他领域概念：

  • 聚合根：一组业务内聚的领域对象的根节点，是数据修改和持久化的唯一入口

    ​ 在代码中，通常就是那个核心的 DO，比如 Order.java

  • 值对象：一个没有唯一标识（ID）的对象，通过其属性值来识别。通常是不可变的

    ​ 例如订单中的 OrderAddress（收货地址）

  • 富血模型：数据和行为封装在一起的对象设计模式，包含业务属性和业务方法，遵循面向对象设计，体现领域核心逻辑

    ​ 符合面向对象思想，业务逻辑内聚，代码可读性和可维护性好，但设计复杂度较高，ORM 映射相对困难，对开发人员要求较高，典型代表是 DO

  • 贫血模型：数据和行为分离的对象设计模式，只包含属性的 getter / setter 方法，业务逻辑在服务层处理，本质是数据载体

    ​ 结构简单，易于理解，便于 ORM 框架映射，开发门槛低，但违背面向对象设计原则，业务逻辑分散在服务层，对象缺乏封装性，典型代表是 PO

项目结构

COLA 通过 Maven 多模块的方式来落地分层架构

层内再按照领域与功能进行分包

1. 适配层：接收请求

2. 应用层：编排业务

OrderServiceImpl 和 OrderCreateCmdExe 协同工作，完成业务用例

3. 领域层：执行核心业务

Order.java 这个聚合根对象，执行所有与创建订单相关的业务规则

4. 基础实施层：提供技术实现

OrderRepositoryImpl 实现了 domain 层定义的接口，负责与数据库交互

通过这个例子，可以清晰地看到每一层都只做了自己该做的事，实现了完美的关注点分离

总结

COLA 架构通过明确的分层和职责划分，将业务复杂性控制在领域层，使得我们的代码：

  • 业务逻辑更清晰：核心业务代码都内聚在 domain 层
  • 可维护性更高：修改数据库或前端框架，不会影响核心业务
  • 可测试性更强：domain 层的纯 Java 代码可以进行快速的单元测试
